# План управления рисками трансформации «Будущее 2.0»

Документ дополняет [карту рисков](risk-map.md): для каждой группы рисков — **меры снижения**, **тип меры** (техническая / управленческая) и **ответственность**.

**Условные обозначения**

| Тип меры | Когда применяется |
|----------|-------------------|
| **Техническая** | Инструменты, архитектура, автоматизация, платформа |
| **Управленческая** | Процессы, роли, обучение, договорённости, финансирование |
| **Смешанная** | Нужны и код/инфраструктура, и организационные решения |

---

## 1. Архитектурные риски

| ID | Меры снижения | Тип |
|----|---------------|-----|
| **A1** | Запретить бизнес-логику в Lakehouse; lakehouse только для хранения и SQL/Spark-витрин. Доменная логика — в микросервисах. Витрина данных — через semantic layer / data products с контрактами. Architecture Decision Records (ADR) + review на Architecture Board. | Смешанная |
| **A2** | Formal Domain Map (медицина, финтех, ИИ, data platform). Запрет shared database между доменами; интеграция — Kafka + published language / API. Отдельные облачные landing zone на домен. | Смешанная |
| **A3** | Технический guardrail: pipeline «только агрегаты» в lakehouse; автоматические сканеры PII/PHI в каталоге (Macie, DLP, custom rules). Data contract: поля med-карт в deny-list для витрины. Периодический audit DPO. | Смешанная |
| **A4** | Паттерн **Strangler Fig**: источник истины — новый сервис; DWH — read replica / CDC только для отчётности до cutover. Reconciliation-джобы и метрики расхождений. Чёткая дата отключения двойной записи на домен. | Техническая |
| **A5** | Стандарты топиков (naming, retention, compaction). Schema Registry (Avro/Protobuf). Обязательные DLQ и runbook. Квоты и мониторинг lag (Prometheus/Grafana). Platform team как владелец Kafka. | Техническая |
| **A6** | Централизованный PAP; политики как код (OPA/Rego). Кэш решений PDP, асинхронное обогащение PIP. Матрица ролей по доменам и регионам; тесты политик в CI. | Смешанная |
| **A7** | Roadmap decommission: Camel/DWH только с ACL и датой sunset. KPI «% трафика через события». Ежеквартальный отчёт Architecture Board. | Управленческая |
| **A8** | Внедрить **Data Catalog + Schema Registry** в первые 6 месяцев (этап 1). Обязательная регистрация новых источников (фарма, IoT) до prod. | Смешанная |
| **A9** | Multi-AZ, cross-region replica для критичных БД и S3. Регулярные DR-учения (RTO/RPO согласованы с бизнесом). IaC для воспроизводимого восстановления. | Техническая |
| **A10** | Partner Integration Gateway: единый вход, события наружу/внутрь, throttling, mTLS. Шаблон onboarding для партнёров (SDK, схемы, sandbox). | Смешанная |

## 2. Организационные риски

| ID | Меры снижения | Тип |
|----|---------------|-----|
| **O1** | Federated governance: у каждого домена свой product owner и data steward, общие стандарты — центр компетенций. SLA на предоставление data products. Спонсорство трансформации на уровне C-level. | Управленческая |
| **O2** | Программа обучения (Kafka, cloud, ABAC, data mesh). Найм/platform team. Партнёрство с облачным интегратором на этап 1–2 с transfer knowledge. | Управленческая |
| **O3** | RACI: владелец витрины данных (Data Platform), владельцы качества — домены. SLA отчётов в договорах между доменом и платформой. Единый backlog «витрины». | Управленческая |
| **O4** | Морatorium на новые point-to-point интеграции в DWH (исключения через Architecture Board). Реестр инициатив и привязка к доменной roadmap. | Управленческая |
| **O5** | Пилоты UX с операторами клиник; поэтапный rollout React UI; гибридный период (read-only в старом UI). Супер-пользователи-евангелисты в клиниках. | Управленческая |
| **O6** | Портфельное управление: runway 70/30 (продукт/платформа). Архитектурные «quality gates» в Definition of Done. | Управленческая |
| **O7** | Architecture Board (еженедельно/раз в 2 недели): review интеграций, ADR, отклонения от C4 To-Be. Чеклист compliance для мед/фин данных. | Управленческая |
| **O8** | Мультиоблачная стратегия только при необходимости; приоритет — один primary cloud + абстракция через K8s/Terraform. Exit strategy в контрактах. | Смешанная |
| **O9** | Вовлечение Legal/DPO на этапе дизайна (privacy by design). DPIA для новых регионов и партнёров. Чеклист до prod. | Управленческая |
| **O10** | Трёхлетний бюджет трансформации с вехами; KPI (time-to-report, uptime, cost per TB). Go/no-go после пилота с метриками, не субъективно. | Управленческая |

## 3. Технологические риски

| ID | Меры снижения | Тип |
|----|---------------|-----|
| **T1** | Поэтапная миграция: CDC (Debezium), checksum, dry-run на копии. Rollback plan. Окна maintenance согласованы с клиниками. Автотесты целостности после cutover. | Техническая |
| **T2** | Lambda/Kappa architecture: batch для истории, stream для свежих данных. Единые watermarks и идемпотентные consumers. Документированные SLA свежести витрин. | Техническая |
| **T3** | Кэш PDP, batch pre-authorization для отчётов, асинхронный PEP где допустимо. Нагрузочное тестирование на критическом пути. | Техническая |
| **T4** | Формализованные правила анонимизации (k-anonymity / псевдонимизация по стандарту). Регрессионные тесты на выборках; периодический pen-test re-ID. Sign-off DPO. | Смешанная |
| **T5** | Kafka as a product: мониторинг, alerting, auto-scaling, playbook DLQ replay. Chaos engineering на non-prod. | Техническая |
| **T6** | Data residency per region (отдельные buckets/кластеры). Geo-routing на уровне DNS/API Gateway. Юридическая карта данных по регионам. | Смешанная |
| **T7** | Security baseline: IAM least privilege, encryption at rest/transit, secrets manager, CSPM. Регулярные сканы и patch management. | Техническая |
| **T8** | GitOps, policy-as-code (OPA/Sentinel), автотесты infra, canary/blue-green для сервисов. | Техническая |
| **T9** | Anti-Corruption Layer вокруг банка; постепенная замена Camel-маршрутов на события. Выделенная команда интеграции финтех-домена. | Смешанная |
| **T10** | OpenTelemetry end-to-end, data lineage (OpenLineage/Marquez), корреляция business event ID. Централизованный SIEM для security-событий. | Техническая |

## 4. Сводка: технические vs управленческие меры

### Преимущественно **технические** (платформа, код, инфраструктура)

| Область | Примеры мер |
|---------|-------------|
| Данные и миграция | CDC, reconciliation, Strangler, Schema Registry, Data Catalog |
| События | Kafka standards, DLQ, idempotent consumers, stream processing |
| Безопасность | ABAC/OPA, encryption, DLP/PII scanning, IAM baseline |
| Надёжность | Multi-AZ/region DR, observability, chaos tests |
| Доставка | GitOps, IaC, canary releases |

### Преимущественно **управленческие** (люди, процессы, решения)

| Область | Примеры мер |
|---------|-------------|
| Организация | Domain owners, RACI, Architecture Board, C-level sponsor |
| Изменения | Change management для операторов, обучение, moratorium на DWH |
| Комплаенс | DPIA, DPO sign-off, privacy by design |
| Портфель | Бюджет на 3 года, 70/30 runway, go/no-go после пилота |
| Партнёры | Onboarding process, контракты SLA data products |

### **Смешанные** (без обоих не сработает)

- Границы доменов (политика + отдельные БД/аккаунты)
- Запрет медданных в витрине (процесс audit + автоматические сканеры)
- Sunset legacy (roadmap + ACL в коде)
- Региональность (юридическая карта + техническая изоляция)

## 5. План реализации по этапам

### Этап 1 (0–6 месяцев) — пилот

| Действие | Закрываемые риски |
|----------|-------------------|
| Запуск Schema Registry, DLQ, базового observability | A8, T5, T10 |
| Пилот в 1–2 доменах (финансы или patient flow) | A4, T1 |
| Architecture Board + ADR-процесс | O7, A2 |
| RACI витрины данных, обучение core team | O2, O3 |
| DPIA для пилотного домена | O9, A3 |

### Этап 2 (6–18 месяцев) — масштабирование

| Действие | Закрываемые риски |
|----------|-------------------|
| ACL для Camel/DWH, потоковые витрины | A7, T9, T2 |
| Data Catalog и data products по доменам | A1, O4 |
| Расширение ABAC на регионы | A6, T6 |
| Change management для Operator UI | O5 |
| DR-учения | A9, T1 |

### Этап 3 (18–36 месяцев) — целевая платформа

| Действие | Закрываемые риски |
|----------|-------------------|
| Decommission DWH/Camel на некритичных потоках | A7, A4 |
| Partner Gateway (фарма, IoT) | A10, T9 |
| Near-real-time витрины, отказ sync на critical path | T2, A2 |
| Мультирегион production | T6, A9 |

## 6. Мониторинг рисков

| Практика | Периодичность | Владелец |
|----------|---------------|----------|
| Реестр рисков (обновление вероятности/влияния) | Ежемесячно | Enterprise Architect |
| Метрики: % событий vs sync, lag Kafka, время отчёта, инциденты P1 | Еженедельно | Platform + домены |
| Architecture compliance review | На каждый major release | Architecture Board |
| Отчёт C-level по трансформации | Ежеквартально | Program Director |

**Ключевые KPI трансформации**

- Time-to-market нового data product (цель: снижение в 3–5 раз к 36 мес.)
- Доля интеграций через события vs legacy-мосты (цель: >80% к 36 мес.)
- RTO/RPO критичных сервисов (согласованы с бизнесом)
- Количество инцидентов, связанных с качеством/доступом данных (тренд вниз)
- Удовлетворённость бизнес-пользователей витриной (опрос, NPS внутренний)

## 7. Эскалация

| Уровень | Условие | Действие |
|---------|---------|----------|
| 1 | Риск средний, в пределах плана | Команда домена + platform |
| 2 | Высокое влияние или срыв вехи этапа | Architecture Board, пересмотр scope |
| 3 | Регуляторный инцидент, утечка данных, простой > SLA | CISO, DPO, CEO IT committee, при необходимости остановка релизов |

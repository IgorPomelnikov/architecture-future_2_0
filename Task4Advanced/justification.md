# Обоснование событийной архитектуры vs Camel ESB и DWH



## Проблема текущего подхода (As-Is)

| As-Is | Последствие для бизнеса |
|-------|-------------------------|
| Бизнес-логика и отчётность в **DWH** | Любое новое направление (банк, фарма, IoT) тянет тяжёлые ETL и месяцы согласований |
| **Camel** как центральная оркестрация | Скрытые зависимости, «спагетти» маршрутов, единая точка отказа |
| Синхронные интеграции и batch в DWH | Отчёты часами; нет near-real-time для операций и витрины |
| Общие таблицы для клиники и финтех | Нарушение доменных границ, риски регуляторики (мед + фин данные) |

---

## Почему выбран событийный подход (To-Be)

Поток **запись → оплата → визит → исследование** на Event Storming естественно разбивается на **четыре bounded context** с обменом **фактами** (розовые стикеры), а не вызовами «в общую базу»:

1. **Booking** публикует `СозданиеСчета` и ждёт `Оплата` — не знает, как устроен банк внутри Payment.
2. **Patient Care** стартует визит по `ПодтверждениеЗаписи`, не опрашивая DWH.
3. **Research** масштабируется отдельно (лаборатория, снимки, ИИ), возвращая только `ПередачаРезультатов`.

Это **choreography**: нет центрального оркестратора на критическом пути, контракт — версионируемое событие.

## Преимущества для «Будущего 2.0»

### Гибкость

- Новый домен (страхование, фарма, кредиты) **подписывается** на `ВизитЗавершен`, `Оплата` без правок Booking/Patient Care.
- Внутри BC можно менять агрегаты и UI (React вместо PowerBuilder), если **контракт события** сохранён.
- Legacy **DWH/Camel** остаются **ACL-мостом** на миграции (CDC → Kafka), а не ядром интеграции.

### Масштабируемость

- **Research** и **Payment** масштабируются независимо от записи на приём (разная нагрузка CPU/IO).
- Kafka партиционируется по `patientId` / `bookingId`; сотни ТБ уходят в доменные хранилища и lakehouse **агрегатами**, не через единый DWH.
- Регионы (США/ЕС/РФ): события и данные остаются в regional topic + regional consumer.

### Скорость реакции

- Оплата → подтверждение записи — **секунды** (событие `Оплата`), а не ночной batch из DWH.
- Результаты исследования → врач: `ПередачаРезультатов` в near-real-time, а не «отчёт соберётся утром».
- Витрина данных для бизнеса строится на **подписке на интеграционные/анонимизированные** потоки, без медкарт в аналитике (требование из task.log).

### Устойчивость

- Временная недоступность Payment не блокирует создание записи: Booking фиксирует ожидание оплаты, **DLQ и retry** на стороне потребителя.
- Сбой Research не откатывает визит: Patient Care видит статус «исследование в процессе».

## Сравнение с Camel и DWH

| Критерий | Camel + DWH | События (Kafka) + DDD |
|----------|-------------|------------------------|
| Связность | Высокая (общие маршруты/таблицы) | Низкая (контракт события) |
| Time-to-market нового продукта | Медленный (ETL, согласование схем DWH) | Быстрее (новый consumer) |
| Операционная нагрузка | Рост маршрутов Camel, деградация SQL 2008 | Платформенная команда Kafka + каталог схем |
| Аналитика | Всё в одном DWH | Операции в BC; аналитика — lakehouse + витрина без PHI |
| Отладка | Сложно проследить цепочку | Correlation `eventId`, `bookingId` в логах и трассировке |

## Риски и как их снимаем

| Риск | Мера |
|------|------|
| «Распределённый монолит» через Kafka | Строгие integration events (см. [events.md](events.md)), Schema Registry, запрет shared DB |
| Дубли и порядок событий | Идемпотентные consumers, ключи партиций, outbox в BC |
| Сложность для команд | Event Storming, единый каталог событий, пилот в Booking + Payment (0–6 мес.) |

---

## Вывод

Событийная архитектура по схеме Event Storming **соответствует** целевой картине «Будущего 2.0»: домены развиваются отдельно, интеграции — через **публикуемые факты**, Camel и DWH — **временные мосты**, а не центр экосистемы. Это даёт измеримый выигрыш в **гибкости подключения банка и новых партнёров**, **масштабировании исследований и ИИ** и **скорости сквозного процесса** «запись — оплата — приём — диагностика».
